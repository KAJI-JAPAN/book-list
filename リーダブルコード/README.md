# 目的
- エンジニアとして自分を含むみんなが読みやすいコードを書けるようにする

## いいコードとは？？
- 短いコードではなく、長くても理解しやすいコード
- コードは他の人が最短で理解しやすいように書く
  - 10秒ではなく、5秒で理解できるように書く

## 名前に情報を詰め込む
- tmpやvalなどの汎用的な命名は使用しない
- ❌ form / error → ⭕️personalForm / ️personalFormErrorなど

## 大きなメソッドの分割をする
- 関数やプログラムは「一つの責務」に限定する 
- 一つの関数・クラス・モジュールが複数の処理や役割を担っていないか常に意識する。 もし複数の責務が含まれている場合は、それらを明示的に列挙し、分割・再構成できないかを検討する。 
- 初期値設定と更新処理を分離する設計を意識する 
- 初期化と更新を同一の処理で行わず、 
  - 初期値を設定する処理 
  - 値を変更・加工する処理 を分けることで、責務が明確になり、テストや再利用が容易になる。 
  - 共通処理は抽出し、意味のある単位にまとめる 
  - ロジックの重複や、概念的にまとまっている処理は関数・メソッドとして切り出す。 
  - 抽出する際は「何をしているか」が名前から直感的に分かることを重視する。 
  - 処理は関数やクラスに閉じ込め、組み合わせで表現する 
低レベルな処理をその場に書き散らさず、関数やクラスに閉じ込めることで、 それらを組み合わせて高レベルな処理を表現できる設計を目指す。 実務では、この「意味のある部品を組み合わせる」構造が保守性・可読性に直結する。

## コードに思いを込める
- コードがやっていることを言葉で説明しなてからその通りにコードを書く
- おばあちゃんでもわかるように説明する
- 口に出すことで思考が整理されるので独り言が大事

## コードはできるだけ書かない
- 開発したい機能に過剰な機能は持たさせない
- 要求をそのまま飲むのではなく、できるだけ簡単に解決できる方法を探す
- 言語の標準APIを読んで自分の引き出しを増やす
